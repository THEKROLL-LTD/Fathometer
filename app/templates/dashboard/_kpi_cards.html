{# Dashboard-KPI-Cards-Grid (Block M, ADR-0020). Fuenf Karten in einem responsiven Grid. Jede Karte ist ein Include von `servers/_kpi_card.html` — derselbe Partial wie auf dem Server-Detail-View (Block K, ADR-0018), erweitert um den optionalen `link_url`-Parameter aus Task #9a. Klick auf eine Karte setzt den entsprechenden URL-Filter und HTMX-swappt das Dashboard-Pane. Variablen-Vertrag: - `quick_stats` : QuickStats — total_open / kev_open / critical_open / high_open / stale_servers (filter-unabhaengige OPEN-Counter, siehe ADR-0020 „Sparkline-Semantik"). - `kpi_sparklines` : dict[str, list[int]] — Keys total/kev/critical/high, jeweils 50 OPEN-Snapshots aelteste zuerst. - `stale_sparkline` : list[int] — 50 Daily-Stale-Server-Counts. Implementierungs-Entscheidung (gegenueber Block-M-Brief): wir nutzen `{% with %}…{% endwith %}` statt `{% include … with context only ... %}`. Jinja2 unterstuetzt `with context` als Modifier, aber keine inline-Key- Value-Parameter-Liste auf `{% include %}`. `{% with %}` ist die portable Variante und exakt das Pattern, das `servers/detail.html` (Block K) bereits fuer dieselben KPI-Card-Includes nutzt — kein Refactor des Card-Partials noetig. #}
